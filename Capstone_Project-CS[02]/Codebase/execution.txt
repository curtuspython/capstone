# ==============================================================================
# CS[02] -- CV Sorting using LLMs
# Execution Instructions
# ==============================================================================
#
# WHAT THIS PROJECT DOES:
#   Takes a job description file and a directory of candidate CV files as
#   input.  Extracts structured requirements from the JD using LLM #1
#   (LangChain + gemini-2.5-flash).  Extracts structured resume data from
#   each CV using pyresparser (Tier 1) or spaCy 3.x NER + regex (Tier 2
#   automatic fallback).  Scores every CV against the JD using LLM #2
#   (LangChain + gemini-2.5-pro) across five dimensions: must-have skills,
#   nice-to-have skills, experience fit, keyword match, and semantic
#   similarity.  Semantic similarity is computed via a four-tier fallback:
#   LlamaIndex GeminiEmbedding -> Gemini SDK text-embedding-004 -> TF-IDF
#   cosine (scikit-learn) -> neutral default 50.0.  Candidates are ranked
#   by a weighted composite score and the full report is printed to the
#   terminal.  An optional --interactive flag enables real-time query
#   refinement without restarting the pipeline.
#
#   Input  : job description file (.txt / .pdf / .docx)
#             directory of candidate CV files (.pdf / .docx / .txt / .md)
#   Output : ranked candidate list printed to the terminal;
#             optional TXT file via 'export' in interactive mode
#
# ==============================================================================
# TWO LLMs  (project requirement: minimum two LLMs)
# ==============================================================================
#
#   LLM #1 : gemini-2.5-flash  (via LangChain ChatGoogleGenerativeAI)
#     Role : Analyses the job description ONCE and extracts structured
#            requirements as a JSON object (title, must_have, nice_to_have,
#            experience_min, keywords, summary).
#     Why  : Flash is fast and cost-efficient for this single-run structured-
#            extraction task.  The schema is fixed and deterministic --
#            accuracy over nuance is not needed here.
#
#   LLM #2 : gemini-2.5-pro  (via LangChain ChatGoogleGenerativeAI)
#     Role : Scores each CV against the JD requirements once per candidate
#            and returns a JSON object with five dimension scores plus
#            narrative feedback (strengths, gaps, recruiter recommendation).
#     Why  : Pro is the most capable Gemini model.  CV scoring requires
#            reading a full CV, inferring skills from described experience,
#            and producing calibrated scores that directly determine the
#            final hiring ranking.  Accuracy here has the highest impact.
#
#   Both models are overridable at runtime:
#     --llm1 <model_name>   (default: gemini-2.5-flash)
#     --llm2 <model_name>   (default: gemini-2.5-pro)
#
# ==============================================================================
# TECHNOLOGY STACK
# ==============================================================================
#
#   Google Gemini API    LLM inference for both JD analysis and CV scoring.
#                        Two separate model instances via llm_client.py.
#
#   LangChain            Prompt orchestration, structured output parsing.
#                        Used in jd_analyzer.py (LLM #1 chain) and
#                        cv_scorer.py (LLM #2 chain) via:
#                          ChatPromptTemplate | ChatGoogleGenerativeAI
#                                            | JsonOutputParser
#
#   LlamaIndex           Semantic matching framework (project requirement).
#                        GeminiEmbedding is attempted first (Tier 1) for
#                        computing cosine similarity between JD and CVs.
#                        Falls back to Gemini SDK (Tier 2) if the
#                        llama-index-embeddings-gemini package returns
#                        HTTP 404 (targets deprecated v1beta endpoint).
#
#   pyresparser (Tier 1) Structured information extraction from PDF/Word
#                        resumes (project requirement).  Attempted first
#                        on every CV.  NOTE: incompatible with spaCy 3.x
#                        (raises [E053] due to outdated config.cfg format);
#                        the error is caught and Tier 2 takes over silently.
#
#   spaCy 3.x (Tier 2)   Automatic fallback for structured resume extraction
#                        via Named-Entity Recognition + regex.  Extracts
#                        name, email, phone, skills, education, degree, and
#                        years of experience.  Compatible with Python 3.11+.
#
#   scikit-learn (Tier 3) TF-IDF cosine similarity used when both LlamaIndex
#                        and Gemini SDK embeddings are unavailable.  Raw
#                        cosine values are batch min-max normalised to
#                        [10, 100] to stay comparable with embedding scores.
#
#   pypdf / python-docx  Raw text extraction from PDF and Word CV files.
#
# ==============================================================================
# VERSIONS
# ==============================================================================
#
#   Python : 3.11 or higher  (tested on 3.13)
#
#   Required packages (all listed in requirements.txt):
#
#     google-genai                  >= 1.0.0

# ========================================================================
# SCORING METHODOLOGY
# ------------------------------------------------------------------------
# Two Large Language Models via the Google Gemini API, orchestrated
# by LangChain with LlamaIndex for semantic matching:
#
#   LLM #1 (gemini-2.5-flash)
#     Analyses the job description once and extracts structured
#     requirements (must-have, nice-to-have, keywords, min experience).
#     Uses a LangChain ChatPromptTemplate -> JsonOutputParser chain.
#
#   LLM #2 (gemini-2.5-pro)
#     Evaluates each CV against those requirements and produces
#     dimension-level scores plus narrative feedback (strengths,
#     gaps, recruiter note).  Uses LangChain chain per candidate.
#
#   Semantic Scoring (4-tier fallback -- first that succeeds wins):
#     Tier 1  LlamaIndex GeminiEmbedding   (cosine similarity)
#     Tier 2  Gemini SDK text-embedding-004 (v1 endpoint, isolated client)
#     Tier 3  TF-IDF cosine similarity      (scikit-learn, local)
#     Tier 4  Neutral default 50.0          (if all above fail)
#
#   Resume Extraction (2-tier fallback):
#     Tier 1  pyresparser  (attempted first; may raise E053 on spaCy 3.x)
#     Tier 2  spaCy 3.x NER + regex  (automatic fallback)
#
#   Composite Score Weights (must sum to 100%):
#     Must-Have      35%
#     Semantic Match 20%  (LlamaIndex / TF-IDF fallback)
#     Experience     20%
#     Nice-to-Have   15%
#     Keyword Match  10%
# ========================================================================

#     Example:
#       [interactive] > show 1

#     Output format:
#       ------------------------------------------------------------
#         RANK #1  -  ALICE JOHNSON

#         File           : ./resumes/alice_resume.pdf

#           Skills     : python, fastapi, django, aws, docker, ...

#           Degree     : Bachelor, Master
#           Experience : 6.0 years
#       ------------------------------------------------------------
#
#   filter <skill>
#     Narrow the ranking to candidates whose raw CV text or structured
#     data contains <skill> (case-insensitive substring match).  The
#     updated ranking table is printed after filtering.
#     Note: does not re-score; simply filters the existing scored set.
#
#     Example:
#       [interactive] > filter kubernetes
#       [interactive] Filtered to 3 candidate(s) matching 'kubernetes'.
#       (ranking table reprinted for the filtered subset)
#
#   reset
#     Remove the active skill filter and restore the full ranking from
#     current_scored.  Respects the current min-score threshold if set.
#
#     Example:
#       [interactive] > reset
#       [interactive] Filters cleared. Full ranking restored.
#
#   min-score <N>
#     Change the minimum composite score threshold on the fly.
#     Candidates below N are kept in the list but flagged [below threshold].
#     The ranking table is reprinted immediately.  Does NOT re-score.
#
#     Example:
#       [interactive] > min-score 80
#       [interactive] Min-score set to 80.0. Re-ranked.
#       (ranking table reprinted with [below threshold] flags)
#
#   edit-must <skill1, skill2, ...>
#     Replace the must-have requirements list with the comma-separated
#     values entered on the same line.  Changes are staged in current_reqs
#     but do NOT affect scores until 'rescore' is run.
#
#     Example:
#       [interactive] > edit-must Python, FastAPI, PostgreSQL, Docker
#       [interactive] Must-have updated to: ['Python', 'FastAPI', 'PostgreSQL', 'Docker']
#       [interactive] Run 'rescore' to re-score with updated requirements.
#
#   edit-nice <skill1, skill2, ...>
#     Same as edit-must but for the nice-to-have requirements list.
#
#   edit-keywords <kw1, kw2, ...>
#     Same as edit-must but for the keywords list used by LLM #2 for
#     keyword-match scoring.
#
#   rescore
#     Re-run LLM #2 (gemini-2.5-pro) on ALL candidates in current_scored
#     using the current requirements (after any edits via edit-must /
#     edit-nice / edit-keywords).  This is the most expensive command:
#     it makes one Gemini API call per candidate.  After re-scoring,
#     current_ranked is rebuilt and the updated ranking table is printed.
#     Use 'rerank' instead if you only changed min-score or filter
#     and do not need new LLM scores.
#
#   rerank
#     Rebuild current_ranked from current_scored without calling the LLM.
#     Re-applies the current filter and min-score threshold.  Fast -- no
#     API calls made.  Use this after changing min-score or filter.
#
#   export
#     Save the current ranking (including any re-scoring or filtering
#     applied during this session) to a timestamped TXT file on disk.
#     The file is written to the directory specified by --output
#     (default: current directory).
#     This is the ONLY way this tool writes any file to disk.
#
#     Filename format:  cv_ranking_report_<YYYYMMDD_HHMMSS>.txt
#
#     Example:
#       [interactive] > export

#     optionally enters the interactive refinement loop.
#
#   llm_client.py
#     Centralised Gemini client initialisation.  Manages:
#       - Default genai.Client (v1beta) for generate_content calls
#       - Dedicated genai.Client (v1) for embed_content calls
#         (isolated instance to avoid LlamaIndex v1beta routing contamination)
#       - LangChain ChatGoogleGenerativeAI instances (cached per model+temperature)
#       - LlamaIndex GeminiEmbedding wrapper (lazy-loaded, Tier 1 semantic)
#     Called once at startup by main.py via llm_client.init(api_key).
#
#   resume_parser.py
#     Two-tier structured resume extraction:
#       Tier 1 -- pyresparser  (project requirement, attempted first)
#       Tier 2 -- spaCy 3.x NER + regex  (automatic fallback)
#     Also handles raw text extraction from PDF (pypdf), Word (python-docx),
#     and plain-text files.  Exposes two public functions used by main.py:
#       extract_text_from_file()         -- raw text from a single file
#       parse_resumes_from_directory()   -- scans a folder, returns candidate list
#
#   jd_analyzer.py
#     Uses LLM #1 (gemini-2.5-flash) via a LangChain chain to extract
#     structured hiring criteria from the job description text:
#       ChatPromptTemplate | ChatGoogleGenerativeAI | JsonOutputParser
#     Returns a dict: title, must_have, nice_to_have, experience_min,
#     keywords, summary.  Runs once per session.
#
#   cv_scorer.py
#     Uses LLM #2 (gemini-2.5-pro) via a LangChain chain to score each
#     candidate CV against the structured JD requirements:
#       ChatPromptTemplate | ChatGoogleGenerativeAI | JsonOutputParser
#     Returns per-candidate scores across five dimensions:
#       overall_score, must_have_score, nice_to_have_score,
#       experience_score, keyword_score, strengths, gaps, recommendation.
#     Called once per candidate inside score_all_cvs().
#
#   ranker.py
#     Computes semantic similarity scores via a four-tier fallback:
#       Tier 1 -- LlamaIndex GeminiEmbedding cosine similarity
#       Tier 2 -- Gemini SDK text-embedding-004 (v1 endpoint)
#       Tier 3 -- TF-IDF cosine similarity (scikit-learn, normalised [10,100])
#       Tier 4 -- Neutral default 50.0
#     Combines all five dimension scores into a weighted composite score:
#       must_have 35% | semantic 20% | experience 20% | nice_to_have 15% | keyword 10%
#     Sorts candidates descending by composite score and assigns 1-based ranks.
#
#   report_generator.py
#     Builds the human-readable report as a list of plain-text lines.
#     print_report()         -- outputs the report directly to the terminal
#     save_report_to_file()  -- writes the same content to a TXT file on disk
#                               (called only by the interactive 'export' command)
