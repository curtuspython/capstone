# ==============================================================================
# CV Sorting using LLMs  -  CS[02]  -  Execution Instructions
# ==============================================================================
#
# WHAT THIS PROJECT DOES:
#   Reads a job description + a folder of candidate CVs, scores every CV
#   using two Google Gemini LLMs orchestrated by LangChain, computes
#   semantic similarity (Gemini embeddings -> TF-IDF cosine fallback),
#   extracts structured resume data via pyresparser (Tier 1) with spaCy
#   3.x NER as the automatic fallback (Tier 2), ranks candidates by a
#   weighted composite score, and prints the full report directly to the
#   terminal -- no CSV or TXT files are created unless the recruiter runs
#   'export' inside interactive mode.
#   Supports an optional terminal-based interactive mode for real-time
#   query refinement.
#
# TECHNOLOGY STACK:
#   Google Gemini API   ->  LLM inference (gemini-2.5-flash + gemini-2.5-pro)
#   LangChain           ->  Prompt orchestration, score aggregation, structured
#                           output parsing (ChatPromptTemplate + JsonOutputParser)
#   LlamaIndex          ->  Semantic matching framework (embedding interface).
#                           Embedding calls go through google-genai SDK because
#                           llama-index-embeddings-gemini targets the deprecated
#                           v1beta endpoint that no longer serves text-embedding-004.
#   pyresparser (Tier 1)->  Structured information extraction from PDF/Word
#                           resumes (project requirement). Attempted first;
#                           NOTE: incompatible with spaCy 3.x (raises E053 due
#                           to outdated config.cfg format). Falls through to Tier 2.
#   spaCy 3.x (Tier 2)  ->  Automatic fallback for structured resume extraction
#                           via NER + regex. Compatible with Python 3.11+.
#   scikit-learn        ->  TF-IDF cosine similarity as semantic scoring Tier 3
#                           (used when Gemini text-embedding-004 returns 404).
#   --interactive mode  ->  Terminal-based query refinement (no GUI required)
#
# TWO LLMs USED (project requirement: minimum two LLMs):
#   LLM #1 : gemini-2.5-flash  ->  Analyses the job description ONCE
#                                   (fast; suitable for single-run structured extraction)
#   LLM #2 : gemini-2.5-pro    ->  Scores each candidate CV in a loop
#                                   (more capable; justified for accuracy-critical scoring)
#
# SUPPORTED FILE FORMATS:
#   Job description : .txt  |  .pdf  |  .docx
#   Candidate CVs   : .txt  |  .pdf  |  .docx  |  .md
#
# ==============================================================================
# VERSIONS
# ==============================================================================
#
#   Python           : 3.11 or higher (tested on 3.13)
#
#   LLM Models (Google Gemini):
#     LLM #1 : gemini-2.5-flash  (job description analysis  -- runs once)
#     LLM #2 : gemini-2.5-pro    (per-CV scoring loop       -- runs per candidate)
#
#   Core Python Packages:
#     google-genai             >=1.0.0   ->  Gemini API client (new SDK)
#     langchain                >=0.3.0   ->  Prompt orchestration & output parsing
#     langchain-google-genai   >=2.0.0   ->  LangChain <-> Gemini integration
#     llama-index-core         >=0.12.0  ->  LlamaIndex semantic framework
#     llama-index-llms-gemini  >=0.4.0   ->  LlamaIndex LLM wrapper for Gemini
#     llama-index-embeddings-gemini >=0.3.0  ->  LlamaIndex embedding wrapper
#     pyresparser              >=1.0.6   ->  Structured resume extraction (Tier 1)
#     nltk                     >=3.8.0   ->  Required by pyresparser
#     spacy                    >=3.0.0   ->  NER fallback extractor (Tier 2)
#       After install: python -m spacy download en_core_web_sm

#     pypdf                    >=6.9.0   ->  PDF resume text extraction
#     python-docx              >=1.1.0   ->  Word (.docx) resume text extraction
#     fpdf2                    >=2.8.0   ->  PDF report generation

# ==============================================================================
# INTERACTIVE MODE  (--interactive)
# ==============================================================================
#
#   Overview
#   --------
#   After the 4-step pipeline finishes and the report is printed, passing
#   --interactive keeps the program alive and drops into a command prompt:
#
#     [interactive] >
#
#   The recruiter can then explore, filter, and refine the results without
#   re-running the entire pipeline.  Three internal state objects drive the
#   loop:
#
#     current_scored  -- all candidates with LLM scores (mutable)
#     current_ranked  -- the current filtered/ranked view (rebuilt on each cmd)
#     current_reqs    -- the live requirements dict (editable inline)
#
#   Commands
#   --------
#
#   show <rank>
#     Print the detailed breakdown for the candidate at position <rank>.
#     Shows dimension scores (must-have, nice-to-have, experience, keyword,
#     semantic), strengths (+), gaps (-), recruiter recommendation, and
#     the structured profile fields extracted by spaCy NER (email, skills,
#     education, years of experience).
#
#     Example:
#       [interactive] > show 1
#
#   filter <skill>
#     Narrow the ranking to candidates whose raw CV text or structured
#     data contains <skill> (case-insensitive substring match).  Useful
#     for quickly screening on a mandatory technology or certification.
#     After filtering, the compact ranking table is reprinted.
#
#     Example:
#       [interactive] > filter kubernetes
#       [interactive] Filtered to 3 candidate(s) matching 'kubernetes'.
#       ... (ranking table reprinted for filtered set)
#
#   reset
#     Remove the active skill filter and restore the full ranking from
#     current_scored (respects current min-score if set).
#
#   min-score <N>
#     Change the minimum composite score threshold on the fly.  Candidates
#     below N are excluded from current_ranked and marked [below threshold]
#     in the table.  Does NOT re-score -- applies immediately.
#
#     Example:
#       [interactive] > min-score 80
#
#   edit-must <skill1, skill2, ...>
#     Replace the must-have requirements list with the comma-separated
#     values entered inline.  Changes are staged but do not affect scores
#     until you run 'rescore'.
#
#     Example:
#       [interactive] > edit-must Python, FastAPI, PostgreSQL, Docker
#
#   edit-nice <skill1, skill2, ...>
#     Same as edit-must but for the nice-to-have list.
#
#   edit-keywords <kw1, kw2, ...>
#     Same as edit-must but for the keyword list used by LLM #2 for
#     keyword-match scoring.
#
#   rescore
#     Re-run LLM #2 (gemini-2.5-pro) on ALL scored candidates using the
#     current requirements (after any edits via edit-must / edit-nice /
#     edit-keywords).  This is the expensive command -- it makes one
#     Gemini API call per candidate.  After re-scoring, current_ranked
#     is rebuilt and the updated table is shown.
#
#   rerank
#     Rebuild current_ranked from current_scored without calling the LLM.
#     Use this after changing min-score or filter when no re-scoring is
#     needed.  Fast (no API calls).
#
#   export
#     Save the CURRENT ranking (including any filters / re-scoring applied
#     during this session) to a TXT file on disk.  The file is written to
#     the directory specified by --output (default: current directory).
#     This is the ONLY way this tool writes any file -- the normal pipeline
#     run prints everything to the terminal and creates no files.
#
#     The filename is:
#       cv_ranking_report_<YYYYMMDD_HHMMSS>.txt
#
#     Example:
#       [interactive] > export
#       [report_generator] Report saved to: ./reports/cv_ranking_report_20260322_200214.txt
#       [interactive] Report saved -> ./reports/cv_ranking_report_20260322_200214.txt
#
#   help
#     Print the short command cheatsheet.
#
#   quit  /  exit  /  q
#     Exit interactive mode and end the program.
#

# ==============================================================================
# SYSTEM ARCHITECTURE
# ==============================================================================
#
#   main.py              ->  CLI entry point; orchestrates the 4-step pipeline;
#                             prints full report to terminal; interactive mode
#   llm_client.py        ->  Centralised Gemini client initialisation:
#                             - default genai.Client (v1beta) for generate_content
#                             - embed genai.Client (v1) for embed_content
#                               (isolated to bypass LlamaIndex v1beta contamination)
#                             - LangChain ChatGoogleGenerativeAI (cached per config)
#                             - LlamaIndex GeminiEmbedding wrapper
#   resume_parser.py     ->  Two-tier resume extraction:
#                             Tier 1: pyresparser (project requirement)
#                             Tier 2: spaCy 3.x NER + regex (automatic fallback)
#                             + raw text extraction via pypdf / python-docx
#   jd_analyzer.py       ->  LLM #1 (gemini-2.5-flash) via LangChain chain
#                             -- structured JD extraction (must-have, nice-to-have,
#                             keywords, min experience, role summary)
#   cv_scorer.py         ->  LLM #2 (gemini-2.5-pro) via LangChain chain
#                             -- per-CV scoring on 4 dimensions + explanation
#   ranker.py            ->  Semantic scoring (4-tier fallback):
#                             Tier 1: LlamaIndex GeminiEmbedding cosine similarity
#                             Tier 2: Gemini SDK text-embedding-004 (v1 endpoint)
#                             Tier 3: TF-IDF cosine similarity (scikit-learn,
#                                     batch-normalised to [10, 100])
#                             Tier 4: neutral default 50.0
#                             + weighted composite score + ranking
#   report_generator.py  ->  Builds report lines; print_report() outputs to
#                             terminal; save_report_to_file() writes TXT on
#                             explicit 'export' in interactive mode only
#
